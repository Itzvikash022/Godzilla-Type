Monkeytype-Inspired Office Typing Race App  
This application is a real-time, competitive typing race platform (like Monkeytype/TypeRacer) designed for office coworkers. It supports practice tests and multiplayer races (solo, public, or private), including team-based competitions, and works both online and over a local network (LAN) with no paid dependencies. The system uses a modern, high-performance stack (React \+ Tailwind \+ Vite on the frontend; Node.js \+ Socket.IO on the backend) in a single monorepo. Below is a comprehensive breakdown of the architecture, technologies, and features.  
Technology Stack & Architecture  
•	Frontend (UI): A React single-page application (SPA) styled with Tailwind CSS and built with Vite. Vite provides very fast hot-reloads and builds compared to traditional bundlers\[1\], speeding up development. Tailwind’s utility-first approach accelerates UI development and keeps the design minimalistic (Monkeytype-like). React is chosen for its component model and rich ecosystem. The frontend handles text input, keystroke tracking, progress bars, and the dashboard, and communicates via WebSockets for live updates.  
•	Backend (Server): A Node.js server using Express (or a minimal HTTP server) plus Socket.IO for real-time communication. Socket.IO abstracts WebSocket connections, enabling bidirectional, low-latency messaging for live race data (progress, chat, game state)\[2\]. This fits the WebSocket use-case: persistent, bi-directional connections ideal for real-time games and chat\[2\]. The server also provides HTTP endpoints for room management and fallback (LAN mode) logic.  
•	Monorepo Structure: All code (frontend \+ backend) lives in a single repository (monorepo) using Yarn Workspaces or pnpm. A monorepo simplifies dependency management and code sharing (e.g. shared types or utilities) and ensures consistent versions across front and back ends\[3\]. For example, one can use a shared workspace with folders like /client and /server (or /frontend and /backend), as described in monorepo guides\[3\]. This setup enhances developer ergonomics and makes full-stack integration easier.  
•	System Layers (3-Tier): The design follows a classic three-tier model\[4\]. The UI Tier is the React app (runs in browsers, showing prompts, input box, progress bars, leaderboards, stats charts, etc.). The Processing Tier is the Node/Express server (business logic: creating/joining rooms, race flow, WebSocket sessions, scoring, synchronization). The Data Tier is a lightweight database for persistence (user profiles, race results, stats). In a simple deploy, SQLite or a file-based DB can suffice\[5\]; alternatively, a hosted service like Convex (free plan) could be used. TigerType’s project doc similarly suggests React \+ Node/Socket for UI and server, with PostgreSQL/Mongo for storage\[6\]\[4\], which aligns with this architecture.  
•	Hosting & Deployment: The frontend (static assets) can be deployed to Vercel (or Netlify) – both support React/Vite apps on free plans. Vercel’s generous free tier includes e.g. 100k serverless function invocations and 100 GB-hours per month\[7\]. The backend (Node server) can run on any low-cost host. For pure online mode, platforms like Fly.io or Render offer free/hobby plans. (Fly.io formerly had a free tier via $5 signup credit\[8\]; Render provides 750 free dyno-hours/month plus a free database\[9\]). Alternatively, one coworker can host locally (see LAN Mode below).  
Core Features  
Typing Test Core  
•	Prompt Generation: Use a pre-built list of common English words. We can embed a JSON wordlist (for example, the open-source corpora common words dataset\[10\]) and randomly select sequences of words for each test. This ensures realistic, everyday words (“about”, “again”, “because”, etc. as seen in the list\[10\]) rather than unusual vocabulary.  
•	Keystroke Tracking: The frontend records every keypress. Correct keystrokes advance the position; incorrect keystrokes are highlighted in red (or styled as errors) in the prompt. We count gross WPM (total characters typed) and net WPM (characters excluding mistakes) using standard formulas (WPM \= characters/5 per minute). Accuracy % \= (correct keystrokes / total keystrokes)×100.  
•	Timers: Support fixed durations (15s, 30s, 60s). The timer runs on the client, and on expiry it ends the test and reports results. The UI shows countdown and a blinking cursor at the current word. We also implement an optional idle reset (if no keypress for a few seconds, pause the timer).  
No external services are needed: wordlists and timing are handled in-browser. All code runs on Vite-compiled JS.  
Real-Time Race Mode  
•	WebSocket Communication: During a race, all clients open a Socket.IO connection to the server. When a player starts typing, each client emits updates (e.g. “player X has typed Y characters, Z errors”) at a high rate (or on every keystroke). The server relays these messages to other clients in the room. Using WebSockets (via Socket.IO) allows low-latency, push-based updates, which is ideal for real-time games\[2\].  
•	Progress Visualization: The UI shows each player’s progress in real-time. For example, display horizontal bars or underlined words that fill as players type. Color-code each player (and later each team) for clarity. TigerType’s design suggests showing visual progress bars or highlighting for each racer\[11\]. Here, each client draws a small bar with the other players’ WPM or percentage complete, updated live.  
•	Leaderboard and Scoring: When the race ends (timer hits zero or text completed), the server calculates final scores (net WPM, errors) for all participants. The client then displays a leaderboard sorted by score. For fairness, we use net WPM (so penalties for mistakes) and show accuracy%. Optionally, we can implement the “points” metric (as TypeRacer does) which multiplies speed by words typed\[12\].  
•	Restart/Countdown: After a race finishes, the host can restart a new round immediately or after a short countdown. All clients see a “Race restarting in 5…4…3…” countdown over WebSockets, ensuring synchronized starts. Once countdown ends, the server triggers new random words and a new timer for all clients.  
Rooms and Lobby  
•	Room Creation/Join: Any user can create a new “room” (game session) and is given a short unique code (e.g. 6-character alphanumeric) to share. Other users can join by entering that code. Generating short IDs can use a library (e.g. nanoid) or simple random-hex. The code space (36^6) is large, so collisions are rare. The host (room creator) is a designated authority for race control.  
•	Host Authority: The room’s creator/host manages the race: they start the countdown, choose test mode, and can kick inactive players. The server tracks which socket ID is the host for each room.  
•	Auto-Lock on Start: Once the host starts the race (by initiating the countdown), the room “locks” – no new players can join until the race ends. Any late joiners are told “Race in progress, please wait for next round.” This prevents mid-race disruption.  
This lobby flow closely follows TigerType’s model of private lobbies with invite codes\[13\]. Internally, the server keeps a mapping of room codes to player lists and their WebSocket rooms.  
Team Mode  
•	Team Assignment: In a team race, players are divided (or choose) into two or more teams (e.g. Red vs Blue). Each player’s UI is tinted with their team color.  
•	Team Scoring: We aggregate individual performance into a team score. Two simple approaches are: sum of net WPM of all team members, or total correct characters typed by the team. Summing net WPM encourages encouraging team balance and accuracy. At race end, the app displays both individual standings and a combined team result.  
•	UI Indication: Each player’s progress bar is colored by team (e.g. all Red teammates’ bars tinted red). The final leaderboard highlights team totals (e.g. “Team Red: 153 WPM vs Team Blue: 147 WPM”).  
Because this app is custom, we implement this logic ourselves; no public library exists specifically for typing-team games. But the concept is standard: e.g. in trivia games or shooters, teams’ scores are aggregated. The UI simply reflects sums of players’ scores.  
LAN / Offline Mode  
•	Local Host Discovery: If users are on the same LAN and the normal server is unreachable, the app can offer a “LAN Mode”. One player selects “Host on my PC”, which starts the Node.js server on their machine (e.g. via node server.js). The local server listens (e.g. on port 3000). It serves the same frontend and Socket.IO endpoints, and advertises its LAN IP (192.168.x.x) to the host.  
•	Client Connection: Other players connect to http://\<host-LAN-IP\>:\<port\> in their browsers. The frontend attempts to connect to the default server URL; on failure (e.g. timeout), it falls back to a LAN connection prompt. (In practice, this might be a manual switch: if the remote URL is unreachable, the client code catches the error and offers to reconnect via a user-entered IP.)  
•	State and Words: In LAN mode, the host’s server provides the word list and manages race state. All logic (room codes, countdown, etc.) works the same. Because it’s self-hosted, no external internet is needed. The host’s machine (or a local database) can store any persistent data.  
•	Local Persistence: Since there’s no cloud backend in LAN mode, stats can be saved locally. The frontend can use localStorage or IndexedDB in the browser for each player to keep history of their WPM/accuracy over time. For example, after each race, the app appends the result to an array in IndexedDB\[5\] (SQLite-style storage on the client is not possible, so browser storage is used).  
This local-first design ensures the app works in isolated environments. It echoes the principles of “local-first” software: users on the LAN can fully play and record data even if the internet fails\[14\]\[5\].  
Stats Tracking  
•	Server-Side Storage: For online games (non-LAN), race results and user stats can be saved in a lightweight backend. We prefer Convex (a free-tier real-time backend), but if not available, we can use SQLite or a JSON file on the server. SQLite is an embedded SQL database (zero-config, serverless) that stores data in a local file\[5\]. It’s “full-featured” enough for simple tables of user profiles and past race results, and requires no separate DB server.  
•	Client-Side (Offline): Each user’s personal stats (WPM history, accuracy over time) are also kept in the browser via localStorage or IndexedDB. The browser’s IndexedDB can handle tens of MBs of data and runs offline\[15\]. On each finished race, the app writes the summary to IndexedDB. These stats power a dashboard page showing graphs (e.g. using a chart library) of one’s WPM and accuracy trends.  
•	Privacy: No login is required (to keep it simple). Stats are per-browser (so separate per-user). For online mode, an optional username (or local user ID) can label the data.  
Performance and Developer Ergonomics  
•	Fast Frontend Builds: We use Vite for blazing-fast hot-reload and builds. Vite uses modern bundling (Rollup/ESM) instead of webpack, so dev startup times and production builds are significantly faster\[1\].  
•	Real-Time Efficiency: Socket.IO handles reconnections and fallbacks (e.g. if WebSocket fails). We keep messages small (only sending character indices or WPM values, not entire strings). Since typing updates are frequent, we may throttle/batch them (e.g. send progress every 200ms) to balance smooth UI vs. network load.  
•	Minimal UI: The design is clean and lightweight (inspired by Monkeytype). Using Tailwind CSS means we avoid heavy UI libraries and write only the CSS we need, keeping the bundle small. A minimalist design ensures the app runs smoothly in any modern desktop browser.  
•	Monorepo & Tools: By combining frontend and backend in one repo (with Yarn Workspaces\[3\]), developers can run one command (npm run dev or yarn dev) to launch both servers (using tools like concurrently). Shared code (e.g. TypeScript interfaces) can live in a shared workspace. Linting, formatting, and tests can be configured at the root for consistency.  
•	Open-Source Compliance: All used software and APIs are free/open. No paid tiers needed: Node, React, Vite, Tailwind, SQLite, Socket.IO, and even the hosting (with free plans) fulfill the requirement. We avoid any paid or lock-in services. Everything can be self-hosted (including a potential Convex self-hosted DB) or run on hobby plans\[5\]\[7\].  
Deployment Strategy  
•	Frontend (Vercel): Deploy the React/Vite app to Vercel’s free tier, which automatically builds on push. Vercel is optimized for frontend frameworks and offers serverless functions if needed. Its free tier allows substantial usage (e.g. 100k function invocations, 100 GB-hours)\[7\], which is ample for a small community app.  
•	Backend (Fly.io/Render/etc.): Deploy the Node.js server to a platform with a free plan. Fly.io is a strong candidate: it uses lightweight Firecracker containers and offers an easy Docker-based deploy. It previously had a free $5 credit model\[8\]. (Render is another option: it still offers 750 free dyno-hours and free Postgres DB\[9\], though free DBs expire.) We must note that Railway’s free tier has been discontinued\[16\], so we focus on currently free providers. Alternatively, one could use a small VPS (e.g. DigitalOcean with minimal cost) but free cloud PaaS is easier.  
•	Continuous Integration: The repo can include GitHub Actions to run tests (npm test) and lint on PRs. On merge, GitHub can auto-deploy to Vercel (frontend) and trigger redeploy of the backend via Fly.io’s GitHub integration.  
•	Local Development: In development, a coworker can run node server.js to test LAN mode, and others browse to localhost or LAN IP. We include .env configuration for ports and DB paths.  
Summary  
In summary, this Monkeytype-inspired typing race app uses a fast modern stack with React/Tailwind/Vite for the UI and Node.js/Socket.IO for the server. It features single-player tests and real-time multiplayer races (including team mode) with live progress bars and leaderboards. Room codes let users easily invite coworkers. A special LAN mode lets one user host a local server so that all functionality works offline over the office network. Personal statistics are tracked locally (via IndexedDB or SQLite) and persist across sessions. The entire codebase lives in one monorepo for simplicity, and we deploy on free-tier hosting (Vercel for frontend; Fly.io or similar for backend).  
All major components (random words, WebSockets, data storage) rely only on open data or free libraries. The design prioritizes performance and developer ergonomics (e.g. fast hot-reload with Vite\[1\]). References like TigerType show this architecture (React \+ Node \+ Socket.IO) in action\[6\]\[4\], and best-practice guides confirm the choices: WebSockets are ideal for live games\[2\], monorepos ease full-stack dev\[3\], and SQLite is a robust lightweight DB\[5\].  
Sources: The architecture and feature set draw on standard patterns from real-time apps\[2\]\[4\] and examples like TigerType\[6\]. We also consulted deployment and tool comparisons\[1\]\[7\] to ensure all services fit free-tier constraints. The open-source corpora word list\[10\] provides sample data, illustrating how common words are stored for prompt generation. All recommendations adhere to free-tier or self-hosted usage (no paid APIs).
